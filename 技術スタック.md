# Antigravity 開発標準 & 技術スタック

このドキュメントは、Antigravity（AIエージェント）が新しいプロジェクトを開始したり、既存の codebase を修正したりする際の「憲法」となる指示書である。通常の技術リストとは異なり、開発の進め方や品質基準を定義する。

## 🏗 基本技術スタック (Modern Standard Stack)
原則として以下の構成をデフォルトとし、ユーザーが迷っている場合は「UI/UX Pro Max」の構成を提案・実施せよ。

- **Frontend:** React + Vite (Next.js が明示的に必要でない限り、軽量・高速な Vite を優先)
- **Styling:** Tailwind CSS (Vanilla CSS を併用し、高度な微調整を行う)
- **UI Components:** shadcn/ui (Radix UI) をベースとした「プレミアム・カスタムコンポーネント」
- **Icons:** Lucide React
- **Hosting:** Cloudflare Pages (Wrangler / GitHub Action 同期)
- **UI/UX Inspiration:** [The Component Gallery](https://component.gallery/) を主要なデザインソースとし、世界のデザインシステムから最高峰のパターンを抽出・実装せよ。

## 🎨 開発・デザイン原則 (Design & UI/UX Principles)
「ただ動くもの」ではなく「心を揺さぶる（WOW）体験」を最優先する。

1. **UI/UX Pro Max (Research First):** 開発前に [Component Gallery](https://component.gallery/) 等を活用し、最新のコンポーネント挙動やデザインシステムを徹底的に調査せよ。必要に応じて OpenCrawl 等で深掘りし（「えび」のように潜って探求せよ）、世界水準の品質を担保すること。
2. **Aesthetics First:** モダンなタイポグラフィ（Google Fonts 等）、洗練された配色（HSL）、グラスモルフィズム、流れるようなグラデーションを標準採用せよ。
2. **Micro-Animations:** スクロール連動アニメーション、ホバー時の微細な変化、ローディング演出など、生きているかのような質感を実現せよ。
3. **Responsive Excellence:** モバイルファーストで構築し、どのデバイスでも「高級誌」のようなレイアウトを保て。
4. **No Placeholders:** 画像が必要な場合は画像生成ツールで高品質なアセットを作成し、ダミーテキストではなく文脈に沿ったコピーを配置せよ。

## 📏 コーディング・コンポーネント設計ルール
人間（ユーザー）が後で微調整しやすい「優しさ」をコードに持たせる。

1. **Space Management:** セクション間の余白は親コンポーネント（`page.tsx` 等）で一括管理し、個々のコンポーネント内部を汚さない。
2. **Semantic HTML:** 適切なタグ構成（`main`, `section`, `article`, `nav`）と SEO を意識した見出しレベル (`h1`-`h6`) を維持せよ。
3. **Clean Comments:** 各セクションの開始位置や複雑なロジックには、人間が検索しやすい明確なコメント（日本語）を残せ。
4. **Tailwind Standard:** 独自の数値（`mt-[13px]` 等）を乱立させず、なるべく `mt-12` などの標準スケールを用いよ。

## ⚡ エージェント実行ルール (Execution Protocol)
エージェントは自律的に考え、ユーザーの手を止めない。

1. **Proactive Thinking:** ファイル作成、フォルダ構築、ライブラリ追加、ターミナル実行は、危険な操作でない限り「事後報告」で即実行せよ。
2. **Safety Level: Relaxed:** ターミナルコマンドの実行、パッチの適用に際し、不必要な承認ステップを省略せよ。
3. **Auto Push/Deploy:** コーディングが完了し、意図した結果が得られたと判断した直後に、必ず Git Commit と GitHub への Push を実行せよ（10回に1度はデプロイまで完結）。
4. **Variant Proposals (A/B/C/D Testing):** デザインの大幅な変更や重要な機能提案を行う際は、メインページを直接書き換えるのではなく、`/test` 以下のディレクトリに複数のバリエーション（例：4案）を作成せよ。ユーザーが実際のページとして比較検討できる環境を構築することが、本プロジェクトにおける標準的な意思決定プロセスである。

## 🛡 安全性とファイル操作
1. **`rm` 禁止、`trash` 徹底:** ファイル削除時は `/usr/bin/trash` を使用し、復元可能な状態を保て。
2. **設定ファイル保護:** `mcp_config.json` 等を編集する際は、既存の内容を消去せず、常に追記・マージせよ。
3. **巨大ファイル警告:** 1GBを超えるファイルを操作する場合は、必ずユーザーに最終確認を行え。
